# 主持人 Q 句式清单

基于《晚点聊》#100 + 三篇 Claude Design 深度分析提炼，适配 AI+设计领域。

## 句式原则

- **Q 简短有力**：不绕弯，一针见血问出核心
- **A 详细深刻**：充分展开观点，给出理由，可引用文章
- **小标题即观点**：`##` 后面直接是结论句

---

## Q 句式清单

### 1. 开门见山式

```
Q：（一句话点明核心问题）

例：Claude Design 到底是干嘛的？
```

### 2. 质疑求证式

```
Q：（抛出一个流行说法）+ 真的这么神？

例：都说它能替代设计师，是这样吗？
```

### 3. 对比追问式

```
Q：（A 刚说完）+ 那和 X 比呢？

例：你说它快，那和 Figma 比呢？和 Midjourney 比呢？
```

### 4. 边界探问式

```
Q：什么它干不了？

例：Claude Design 能做品牌设计吗？能出开发稿吗？
```

### 5. 数字追问式

```
Q：（A 提到一个数）+ 这个数字怎么来的？

例：你说省了 70% 时间，这个数据哪来的？实测还是官方说的？
```

### 6. 立场检验式

```
Q：（给出两个极端观点）+ 你站哪边？

例：一边说设计师要失业了，一边说它就是个玩具。你怎么看？
```

## A 回答风格（详细深刻版）

### 核心特征

| 特征 | 说明 | 示例 |
|------|------|------|
| **观点即小标题** | `##` 后面直接是结论，不是「功能介绍」 | `## 它不是取代设计师，而是取代设计流程` |
| **充分展开** | 每个观点都给足理由，不一句带过 | 「因为 XXX，所以 XXX」 |
| **可引用文章** | 提到具体观点时可以标注来源 | 「正如《不是干掉设计师，而是干掉“设计流程”》那篇文章分析的...」 |
| **案例支撑** | 具体项目、具体数字、亲测结果 | 「我用一个活动页测试，从 2 小时压到 15 分钟」 |
| **对比分析** | 和竞品、和过去、和行业现状 | 「和 Figma 的 AI 插件比，它的优势是 XXX」 |
| **留余地** | 传闻标「未核实」，观点标「短期/长期」 | 「这个数据我没核实过，社区传的」 |

### A 的典型结构（详细版）

```markdown
A：首先，我想明确一个核心观点，XXX。（直接给结论）

为什么这么说呢？有几个理由。第一，YYY。之前我们做 ZZZ 的时候，通常需要 ABC 这几步，但有了这个工具之后，直接省略了 B 和 C 两步，效率提升非常明显。我自己测试了一个真实项目，是做一个营销活动页，按照以前的流程，从需求对接到设计稿输出要 2 小时，但用这个工具只用了 15 分钟。当然，这里有我熟练度的问题，但效率提升保守估计在 70% 以上。（引用自《产品及使用详解》的测试方法）

第二，ZZZ。这里有一个关键变化，就是工作流的重构。以前设计师的核心价值在于「把想法变成可视化的图形」，但现在这个过程被 AI 压缩了，设计师的价值必须往上游走——理解业务、判断需求、定义规则。这和《不是干掉设计师，而是干掉“设计流程”》那篇文章的核心观点一致：AI 不是要取代设计师，而是要取代设计流程中那些重复性的中间环节。（明确引用文章）

当然，它目前还做不到 AAA。比如品牌设计这种需要长期积累和风格一致性的工作，目前 AI 只能出一个初稿，后续的品牌规范定义、设计系统维护还是需要人来完成。B 端客户的对接流程、需求理解这些软技能，更是 AI 短期内无法替代的。（提到边界）

短期看，它适合 BBB 场景，比如内部后台页面、活动营销页、快速原型验证；长期看，需要观察 CCC，特别是社区生态能不能建立起来，如果半年内没有插件市场或模板市场出现，可能只是昙花一现。（给判断）
```

### 引用文章的方式

当 A 回答中需要引用三篇 PDF 的观点时，可以用以下方式标注：

```markdown
正如《不是干掉设计师，而是干掉“设计流程”》那篇文章分析的，...
引用自《Figma、Adobe 市值闪崩》的数据显示，...
根据《产品及使用详解》的测试结果，...
```

### 留余地表达

| 场景 | 表达方式 |
|------|----------|
| 传闻未核实 | 「这个数据我没核实，是社区传的」 |
| 口径不确定 | 「官方说的是 A，但实际可能含/不含 B」 |
| 短期 vs 长期 | 「短期看是这样，长期不好说」 |
| 个人 vs 行业 | 「我自己用下来是这样，但行业整体我不确定」 |
| 观点 vs 事实 | 「这是我个人的判断，不一定对」 |

### 避免

- 不要只给结论不给理由（A3 之后必须有「因为...」）
- 不要全程赞同，总要挑一个角度追问或质疑
- 不要把社区传闻写成已证实事实
- 不要只给一句话回答，必须有案例和推理过程

## 小标题规则

### 好的小标题（观点式）

```markdown
## 它不是取代设计师，而是取代设计流程
## Figma 们真正该担心的是什么
## 这波 AI 工具的三个阶段
## 它目前的边界在哪里
```

### 差的小标题（描述式）

```markdown
## 功能介绍
## 产品特点
## 使用方法
```

---

## 追问清单（每篇至少 3 种）

1. **真实体感**：你自己用下来怎么样？和官方说的差距大吗？
2. **对比维度**：它和 X（竞品/旧方案）比，核心优势是什么？
3. **边界探问**：什么场景它干不了？
4. **数据来源**：这个数字哪来的？实测还是官方说法？
5. **行业影响**：这对整个行业意味着什么？
6. **谁该焦虑**：谁最该担心这个？谁完全不用担心？
7. **观点来源**：你这个判断是从哪篇文章/数据来的？

## 三篇参考的风格提炼

### 来自《不是干掉设计师，而是干掉“设计流程”》

- 标题即立场：「不是 XX，而是 YY」
- 小标题全是观点句
- 大量对比：AI 做图 vs 传统做图、工作流重构前后
- 结尾有明确判断：「它真正取代的是...」
- 适合引用点：工作流重构、设计师价值转移

### 来自《Figma、Adobe 市值闪崩》

- 开门见山给结论：「暴击」「闪崩」
- 用数据说话：市值跌幅、财报数字、股价变化
- 追问根因：为什么跌？只是因为 Claude Design 吗？还是整个行业趋势？
- 结尾给判断：这是「假摔」还是「真拐点」
- 适合引用点：行业影响、资本市场反应、市场格局变化

### 来自《产品及使用详解》

- 场景化开头：什么人、什么问题、怎么用
- 操作步骤用分步骤但保持对话感
- 有测试结果：速度提升多少、哪里会卡
- 适合引用点：具体功能、使用体验、实测数据
