# 边界情况怎么处理

写 `*-article.md` 时若遇到下列情况，按右列处理，避免幻觉或格式漂移。

| 情况 | 建议 |
|------|------|
| `key_facts` 很少或很模糊 | 仍用对话完成结构；**明确写「公开信息有限」**；少写产品细节、多写**验证问题与工作流边界**；禁止编造未出现的定价/功能。 |
| `notes` 与 `key_facts` 冲突 | **以 key_facts + 可核验来源为准**；`notes` 仅作「有讨论认为…」并标不确定。 |
| 人工选题与 `article_pick` 不一致 | 在文首或编辑说明**显式标注**「本文主题与当日脚本选题不同及原因」，避免口播与日报 silently 脱节。 |
| 用户只要求润色中段 | **保留**原有 `Q：` / 嘉宾名 / `##` 结构；不擅自改成散文或列表简报。 |
| 必须提及竞品但 processing 未列 | 可写「常见对比会提到 X」，**避免捏造**具体版本、价格、评测分数。 |
| 口播 / TTS 可读性 | 少叠括号与过长英文串；必要时中文先出、括号补英文专名。 |
| 篇幅写超或写短 | 目标仍 **1500–4000 字**；超则删重复轮次，短则补「落地/付费/不适用人群」类追问，勿灌水。 |

**自检一句**：若删掉所有 `Q：` 行，稿子里是否还剩「像双人播客」的结构？若不像，多半体裁已跑偏。
